# Защита лабораторной работы №2

## Тема работы

Локализация WPF-приложения тремя разными способами:

1. через `RESX`-ресурсы;
2. через `XAML ResourceDictionary`;
3. через внешнюю библиотеку классов.

Эта лабораторная работа является надстройкой над предыдущей: за основу взято приложение из ЛР №1 про привязки в WPF, а затем в него была добавлена динамическая поддержка двух языков: русского и английского.

---

## Цель работы

Изучить и сравнить три подхода к локализации WPF-приложения, реализовать динамическое переключение языка без перезапуска программы и понять, в каких случаях какой способ удобнее применять.

---

## Что требовалось по ТЗ

По заданию из файла `ЛР_2.md` нужно было:

- добавить в приложение поддержку двух языков: русского и английского;
- сделать переключение языка во время работы программы;
- перевести статические текстовые элементы интерфейса;
- перевести как минимум одно текстовое сообщение, выводимое в процессе работы;
- реализовать это тремя разными способами;
- оформить каждый вариант как отдельный проект.

В решении это выполнено тремя проектами:

- `WpfLocalizationResx` — локализация через `RESX`;
- `WpfLocalizationResourceDictionary` — локализация через словари ресурсов `XAML`;
- `WpfLocalizationExternalLibrary` — локализация через внешнюю библиотеку `LocalizationLibrary`.

---

## Общая идея реализации

Во всех трёх проектах логика приложения осталась одинаковой, изменился только способ хранения и получения переводов.

Общая схема работы такая:

1. В `MainWindow.xaml` находится переключатель языка (`ComboBox`).
2. При выборе языка вызывается обработчик `LanguageComboBox_SelectionChanged`.
3. Обработчик передаёт код языка (`ru` или `en`) в `LocalizationService`.
4. `LocalizationService` меняет текущий источник строк.
5. Через `INotifyPropertyChanged` интерфейс получает уведомление и обновляет надписи.

То есть архитектурно подход одинаковый, а различие только в том, откуда именно берутся строки перевода.

## Способ 1. Локализация через RESX

### Что это такое

`RESX` — это стандартный механизм .NET для хранения локализуемых строк и других ресурсов. Для каждого языка создаётся отдельный файл ресурсов.

В моём проекте используются:

- `Strings.resx`
- `Strings.ru.resx`

`ResourceManager` получает строку по ключу и по выбранной культуре.

### Как реализовано

Основная логика находится в файле:

- `WpfLocalizationResx/Services/LocalizationService.cs`

Там создаётся `ResourceManager`:

```csharp
_rm = new ResourceManager("WpfLocalizationResx.Resources.Strings", typeof(LocalizationService).Assembly);
```

Текущий язык хранится в поле:

```csharp
private CultureInfo? _culture;
```

При смене языка вызывается:

```csharp
public void SetLanguage(string languageCode)
{
    _culture = new CultureInfo(languageCode);
    CultureInfo.CurrentUICulture = _culture;
    CultureInfo.DefaultThreadCurrentUICulture = _culture;
    OnPropertyChanged("");
}
```

Строки выдаются через свойства, например:

```csharp
public string WindowTitle => _rm.GetString("WindowTitle", _culture) ?? "WPF Bindings Demo";
```

### Как это подключено к интерфейсу

В `MainWindow.xaml` и во вкладках используются привязки вида:

```xml
Text="{Binding Localization.WindowTitle}"
```

Например:

- `WpfLocalizationResx/MainWindow.xaml`
- `WpfLocalizationResx/Views/Tabs/DefaultBindingView.xaml`

А в `MainViewModel.cs` сервис локализации пробрасывается в XAML так:

```csharp
public LocalizationService Localization => LocalizationService.Instance;
```

### Плюсы

- стандартный подход платформы .NET;
- удобно хранить строки по ключам;
- хорошо подходит для крупных приложений;
- легко масштабируется на много языков.

### Минусы

- визуально менее удобно редактировать, чем XAML-словарь;
- для WPF иногда требует дополнительной обвязки для динамического обновления интерфейса;
- ключи и свойства приходится поддерживать отдельно.

## Способ 2. Локализация через XAML ResourceDictionary

### Что это такое

Здесь переводы хранятся в XAML-словарях ресурсов. Для каждого языка есть свой файл словаря:

- `Strings.ru.xaml`
- `Strings.en.xaml`

Каждая строка хранится как ресурс с ключом:

```xml
<sys:String x:Key="LanguageSelector">Язык:</sys:String>
```

### Как реализовано

Основная логика находится в:

- `WpfLocalizationResourceDictionary/Services/LocalizationService.cs`

При запуске приложения по умолчанию подключается русский словарь через `App.xaml`:

```xml
<ResourceDictionary Source="Resources/Strings.ru.xaml"/>
```

При смене языка сервис:

1. создаёт новый `ResourceDictionary`;
2. удаляет старый словарь локализации из `Application.Current.Resources.MergedDictionaries`;
3. добавляет новый;
4. вызывает `OnPropertyChanged("")`.

Ключевой метод:

```csharp
private void LoadDictionary(string languageCode)
{
    var uri = new Uri($"Resources/Strings.{languageCode}.xaml", UriKind.Relative);
    var dict = new ResourceDictionary { Source = uri };

    var oldDict = Application.Current.Resources.MergedDictionaries
        .FirstOrDefault(d => d.Source?.OriginalString.Contains("Strings.") == true);

    if (oldDict != null)
        Application.Current.Resources.MergedDictionaries.Remove(oldDict);

    Application.Current.Resources.MergedDictionaries.Add(dict);
}
```

### Как это подключено к интерфейсу

В XAML используются `DynamicResource`, например:

```xml
Text="{DynamicResource DefaultBindingTitle}"
```

Это видно в:

- `WpfLocalizationResourceDictionary/Views/Tabs/DefaultBindingView.xaml`

Также часть строк доступна через сервис и свойство `Localization`, например в `MainWindow.xaml`.

### Плюсы

- очень естественно выглядит именно для WPF;
- удобно использовать прямо в XAML;
- можно менять словарь на лету;
- хорошо сочетается со стилями, шаблонами и темами.

### Минусы

- ключи ресурсов нужно контролировать вручную;
- если ресурсов очень много, словари становятся громоздкими;
- логика локализации сильнее привязана к WPF.

## Способ 3. Локализация через внешнюю библиотеку классов

### Что это такое

В этом варианте переводы вынесены в отдельную библиотеку `LocalizationLibrary`. Это уже не просто файлы ресурсов, а полноценный набор классов.

Смысл подхода:

- есть интерфейс `IStrings`, который описывает все доступные строки;
- есть реализация `StringsRu` для русского языка;
- есть реализация `StringsEn` для английского языка;
- сервис выбирает нужную реализацию в зависимости от языка.

### Как реализовано

Основные файлы:

- `WpfLocalizationExternalLibrary/LocalizationLibrary/IStrings.cs`
- `WpfLocalizationExternalLibrary/LocalizationLibrary/StringsRu.cs`
- `WpfLocalizationExternalLibrary/LocalizationLibrary/StringsEn.cs`
- `WpfLocalizationExternalLibrary/LocalizationLibrary/LocalizationService.cs`

Интерфейс `IStrings` задаёт контракт: какие строки должны быть в каждом языке.

Пример идеи:

```csharp
public interface IStrings
{
    string WindowTitle { get; }
    string TabDefault { get; }
    string LanguageSelector { get; }
}
```

Русский и английский классы реализуют этот интерфейс и возвращают конкретные строки.

Переключение языка в `LocalizationService.cs` работает так:

```csharp
_strings = languageCode switch
{
    "en" => new StringsEn(),
    "ru" => new StringsRu(),
    _ => new StringsRu()
};
```

После этого вызывается:

```csharp
OnPropertyChanged("");
```

### Как это подключено к интерфейсу

Во ViewModel есть свойство:

```csharp
public LocalizationService Localization => LocalizationService.Instance;
```

А в XAML надписи привязаны так:

```xml
Text="{Binding Localization.DefaultBindingTitle}"
```

Это видно в:

- `WpfLocalizationExternalLibrary/WpfApp/MainWindow.xaml`
- `WpfLocalizationExternalLibrary/WpfApp/Views/Tabs/DefaultBindingView.xaml`

### Плюсы

- самая явная и типобезопасная архитектура;
- удобно переиспользовать библиотеку в нескольких приложениях;
- легко тестировать;
- структура понятна с точки зрения ООП.

### Минусы

- больше всего кода;
- при добавлении новой строки нужно менять интерфейс и все языковые классы;
- для простых приложений может быть избыточно.

---

## Главное различие трёх способов

### RESX

Переводы лежат в `.resx`-файлах, а получение строк идёт через `ResourceManager` и `CultureInfo`.

### ResourceDictionary

Переводы лежат в XAML-словарях, а интерфейс получает их через `DynamicResource` и замену словаря ресурсов во время работы.

### Внешняя библиотека

Переводы лежат в обычных C#-классах, а переключение языка означает замену объекта с набором строк.

---

## Сравнение подходов

| Критерий | RESX | ResourceDictionary | Внешняя библиотека |
|---|---|---|---|
| Где хранятся строки | `.resx` | `.xaml` | `.cs` |
| Основной механизм | `ResourceManager` | `MergedDictionaries` + `DynamicResource` | интерфейс + классы |
| Удобство для WPF UI | среднее | высокое | высокое |
| Типобезопасность | средняя | низкая | высокая |
| Простота расширения | высокая | средняя | средняя |
| Переиспользование между проектами | среднее | низкое | высокое |
| Объём кода | средний | средний | высокий |

## Как сделано динамическое переключение языка

Во всех трёх проектах логика одинаковая.

В `MainWindow.xaml` есть `ComboBox` выбора языка:

```xml
<ComboBox SelectedIndex="0"
          SelectionChanged="LanguageComboBox_SelectionChanged">
    <ComboBoxItem Content="{Binding Localization.LanguageRussian}" Tag="ru"/>
    <ComboBoxItem Content="{Binding Localization.LanguageEnglish}" Tag="en"/>
</ComboBox>
```

В `MainWindow.xaml.cs` находится обработчик:

```csharp
private void LanguageComboBox_SelectionChanged(object sender, SelectionChangedEventArgs e)
{
    if (sender is ComboBox combo && combo.SelectedItem is ComboBoxItem item && item.Tag is string langCode)
    {
        LocalizationService.Instance.SetLanguage(langCode);
    }
}
```

После этого сервис локализации уведомляет интерфейс через `INotifyPropertyChanged`, и привязанные элементы обновляются без перезапуска окна.

## Как соблюдён MVVM

Требование MVVM соблюдено следующим образом:

- бизнес-логика и состояние элементов находятся во `ViewModel`;
- представление (`XAML`) только отображает данные и использует привязки;
- локализация подаётся в интерфейс через свойство `Localization`;
- при смене языка `ViewModel` подписана на событие `PropertyChanged` сервиса локализации и вызывает `OnPropertyChanged("")`.

Пример:

```csharp
public MainViewModel()
{
    LocalizationService.Instance.PropertyChanged += (s, e) =>
    {
        OnPropertyChanged("");
    };
}
```

Это есть, например, в:

- `WpfLocalizationResx/ViewModels/MainViewModel.cs`
- `WpfLocalizationResourceDictionary/ViewModels/AllViewModels.cs`
- `WpfLocalizationExternalLibrary/WpfApp/ViewModels/AllViewModels.cs`

## Какой способ лучше и в каком случае

### Если приложение обычное .NET и нужна классическая локализация

Лучше использовать `RESX`.

Почему:

- это стандартный механизм платформы;
- хорошо работает в больших проектах;
- удобен для масштабирования по культурам.

### Если приложение в основном строится вокруг XAML и WPF

Лучше использовать `ResourceDictionary`.

Почему:

- удобнее прямо в разметке;
- хорошо подходит для динамической подмены ресурсов;
- естественно сочетается со стилями и темами.

### Если нужна переиспользуемая архитектура или несколько приложений

Лучше использовать внешнюю библиотеку.

Почему:

- локализацию можно вынести в отдельный модуль;
- удобно тестировать и повторно использовать;
- подход хорошо демонстрирует ООП и архитектурное разделение.

## Вывод по работе

В лабораторной работе реализована одна и та же задача тремя способами. Это позволило сравнить:

- стандартный подход .NET (`RESX`);
- WPF-ориентированный подход (`ResourceDictionary`);
- архитектурный подход через отдельную библиотеку классов.

Практически все три способа рабочие, но они удобны в разных сценариях:

- `RESX` — наиболее стандартный;
- `ResourceDictionary` — наиболее удобный именно для WPF-интерфейса;
- внешняя библиотека — наиболее гибкая и архитектурно чистая.

## Короткие ответы для защиты

### Что такое локализация

Локализация — это адаптация приложения под разные языки и культуры без изменения основной логики программы.

### Что было сделано в этой работе

Я взял приложение из предыдущей лабораторной и реализовал динамическое переключение русского и английского языка тремя способами.

### Почему именно три проекта

Потому что по заданию нужно было сравнить три разных механизма локализации, и удобнее всего это показать на отдельных проектах с одинаковой бизнес-логикой.

### В чём основное отличие этих методов

Отличается место хранения переведённых строк и механизм их подстановки в интерфейс:

- `RESX` — ресурсы .NET;
- `ResourceDictionary` — ресурсы XAML;
- внешняя библиотека — обычные классы C#.

### Как происходит смена языка без перезапуска

При выборе языка в `ComboBox` вызывается метод `SetLanguage`, затем сервис локализации посылает уведомление `PropertyChanged`, и интерфейс обновляет привязанные значения.

### Какой способ я бы выбрал для WPF

Для чистого WPF-интерфейса я бы выбрал `ResourceDictionary`, потому что он лучше всего интегрируется с XAML и позволяет удобно менять ресурсы во время работы.

### Какой способ самый архитектурный

Внешняя библиотека, потому что локализация отделена в самостоятельный модуль с интерфейсом и реализациями.

### Какой способ самый стандартный для .NET

`RESX`, потому что это встроенный механизм платформы .NET для работы с локализуемыми ресурсами.
